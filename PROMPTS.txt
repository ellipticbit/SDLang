/plan We need to create library that can both parse and emit SDLang (SDL for short), a document language similar to XML. The specification can be found here: https://github.com/dlang-community/SDLang-D/wiki/Language-Guide with additional test cases and guidance here: https://sdlang.org/ . The overall design goal is library that broadly replicates the capabilities of System.Text.Json. AOT support is out of scope for this iteration of the library.

The core building block of SDL is a Tag, which is analagous to the System.Text.Json JsonElement type.

In addition to basic parsing/emitting of Tags, we would like to add object serialization/deserialization capabilities to this library. If it would be more performant you may implement a separate parser/emitting for Tags and Object graphs. You should provide both synchronous and async methods for serialization/deserialization. Metadata Attributes should be created similar to System.Text.Json that allow for runtime serialization/deserialization. Tag would be treated like JsonElement and used when no type is found (Object types). This library should also follow the same rules for Generics that System.Text.Json  We should support as many of the JsonSerializerOptions as possible, including object references and support for pretty-printed or unformatted outputs (WriteIndented in JsonSerializerOptions).

There will be ample opportunity for performance optimizations to be applied to this codebase using the new language features of C#/.NET 10 such as spans. These opportunities should be thoroughly explored. Additionally, because this a library that can process uncontrolled user input from untrusted sources, this library needs to be hardened against any potential vulnerabilities in it's input parsing. Of particular note, SDL is UTF8 only, while .NET strings are UCS-2/UTF-16, we would prefer to avoid using .NET strings where possible to avoid the overhead of converting to/from UTF-16.

There should be ample tests of both the basic Tag serialization/deserialization as well as object serialization/deserialization. Include tests for UTF-8 characters like emoji's and Kanji characters.

Lastly, we need to generate the appropriate project properties for this project to be hosted on nuget.org, including the icon.png in the root folder and a markdown README.md for this project. The readme should include a brief description of the project, and the standard information typically included in a README for an open source project, how to add the library to a project, an appropriate number of example of how to integrate (typically via DI) and use the library, and contribution guidelines that include a note saying that any LLM based contributions must include the Prompt used to generate the contribution in the PROMPTS.txt file. The license is BSL-1.0.

Now we need to create a VSIX project for Visual Studio 2026 (2022 support if possible) that allows Visual Studio to support SDLang as a complete language similar to it's support for JSON and XML. This will be a complex project so plan accordingly. This project should live in the vsix folder, not the src folder, and have the following components.

1. For syntax highlighting we want to use a TextMate grammer.
2. We need to add a Language Config for basic SDLang features like bracket matching/auto-close, string quotes matching/auto-close, indentation rules, and any other features of SDLang that can be covered with Language Config.
3. We need to add an internal LSP (this LSP should be lightweight rendering an external process unnecessary) that enables robust error detection, by displaying error squiggles and adding errors to the error list. Note that the parser in EllipticBit.SDLang is strict and will throw on the first error when parsing the document. For the LSP use case it may be necessary to add a permissive mode to the parser to allow it to continue parsing and displaying errors.
4. The LSP should also be extensible via MEF contracts to allow other VSIX plugins to enable semantic validation for their documents. For example, dub.sdl from the D programming languages package manager DUB, has specific tags and valid attribute values that should be evaluated and made available in IntelliSense. And validation errors should likewise be published to the Errors list window via the LSP.
5. Content Type Targeting should be implemented for the "sdlang" target type.
6. Implement a Brokered Service to enable legacy plugins, such as the Visual D plugin to leverage these capabilities for it's dub.sdl support. It is OK for this to be a 2026 only feature if 2022 does not support Brokered services.
7. Create a comprehensive Wiki of Markdown files describing the integration endpoints and API's for MEF/Content Type Targeting/Brokered Service implementations to make it easier for other plugin implementors to integrate with SDLang. This repo is published to both GitLab and GitHub, so the wiki should work on both platforms if possible.
8. Public API endpoints should have the appropriate XML documentation comments attached to them.
