/plan We need to create library that can both parse and emit SDLang (SDL for short), a document language similar to XML. The specification can be found here: https://github.com/dlang-community/SDLang-D/wiki/Language-Guide with additional test cases and guidance here: https://sdlang.org/ . The overall design goal is library that broadly replicates the capabilities of System.Text.Json. AOT support is out of scope for this iteration of the library.

The core building block of SDL is a Tag, which is analagous to the System.Text.Json JsonElement type.

In addition to basic parsing/emitting of Tags, we would like to add object serialization/deserialization capabilities to this library. If it would be more performant you may implement a separate parser/emitting for Tags and Object graphs. You should provide both synchronous and async methods for serialization/deserialization. Metadata Attributes should be created similar to System.Text.Json that allow for runtime serialization/deserialization. Tag would be treated like JsonElement and used when no type is found (Object types). This library should also follow the same rules for Generics that System.Text.Json  We should support as many of the JsonSerializerOptions as possible, including object references and support for pretty-printed or unformatted outputs (WriteIndented in JsonSerializerOptions).

There will be ample opportunity for performance optimizations to be applied to this codebase using the new language features of C#/.NET 10 such as spans. These opportunities should be thoroughly explored. Additionally, because this a library that can process uncontrolled user input from untrusted sources, this library needs to be hardened against any potential vulnerabilities in it's input parsing. Of particular note, SDL is UTF8 only, while .NET strings are UCS-2/UTF-16, we would prefer to avoid using .NET strings where possible to avoid the overhead of converting to/from UTF-16.

There should be ample tests of both the basic Tag serialization/deserialization as well as object serialization/deserialization. Include tests for UTF-8 characters like emoji's and Kanji characters.

Lastly, we need to generate the appropriate project properties for this project to be hosted on nuget.org, including the icon.png in the root folder and a markdown README.md for this project. The readme should include a brief description of the project, and the standard information typically included in a README for an open source project, how to add the library to a project, an appropriate number of example of how to integrate (typically via DI) and use the library, and contribution guidelines that include a note saying that any LLM based contributions must include the Prompt used to generate the contribution in the PROMPTS.txt file.
